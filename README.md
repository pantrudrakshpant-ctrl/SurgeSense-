# ⚡ Real-Time Dynamic Demand Forecasting & Intent-Based Pricing Engine

A **local-first**, dependency-light ML system that forecasts short-horizon
product demand and computes bounded, elasticity-driven dynamic prices —
entirely from first-principles NumPy math and a lightweight Scikit-Learn
ensemble, reacting to a live simulated click-stream in real time.

No cloud pipeline, no message broker, no Docker Compose stack. Just Python,
NumPy, Scikit-Learn, and FastAPI — because the goal of this project is to
prove mastery of the underlying mathematics, not to wire together managed
services.

---

## 🧠 Why this architecture

Real e-commerce demand has two very different behaviors layered on top of
each other:

1. A **smooth, slow-moving baseline** — seasonality, day-of-week habits,
   long-run trend. Best modeled with a classical statistical smoother:
   cheap, interpretable, and stable even with little data.
2. A **sharp, non-linear reaction** to live signals — a competitor price
   drop, a click-stream spike, urgency from low stock. Tree ensembles
   excel here because they capture *interaction effects* (e.g. "high
   add-to-cart rate **and** low stock" behaves differently than either
   signal alone) without hand-engineering every cross term.

So instead of a single model, this engine runs a **two-branch ensemble**
and blends the branches with a data-driven, adaptive weight — see
[`model.py`](./model.py) for the full derivation.

```
final_demand = w · ExponentialSmoothing(history)   +   (1 − w) · GradientBoosting(live_features)
               \_______________________/                \_____________________________/
                     stable baseline                         reactive, non-linear signal

   w = confidence-adjusted blend weight, derived from live traffic volume
       via a logistic squashing function (more live signal → less w)
```

## 🔄 Live Dataflow

```
┌─────────────────────────────┐
│   ECommerceEventGenerator   │   background daemon thread (threading.Thread)
│   (simulator.py)             │   emits page_view / add_to_cart / purchase
│                              │   + drifts stock_level & competitor_price
└──────────────┬───────────────┘
               │ push()
               ▼
┌─────────────────────────────┐
│  ThreadSafeSlidingWindow     │   deque(maxlen=100) guarded by threading.Lock
│  (last 100 live events)      │   → compute_demand_velocity() on every read
└──────────────┬───────────────┘
               │ live_signals (page_views, add_to_carts, demand_velocity)
               ▼
┌───────────────────────────────────────────────────────────┐
│                     DemandForecaster                       │
│  ┌───────────────────────┐      ┌───────────────────────┐ │
│  │ ExponentialSmoother    │      │ GradientBoosting       │ │
│  │ (pure NumPy, Holt's    │      │ Regressor (sklearn)    │ │
│  │  level + trend)        │      │ on engineered features │ │
│  └───────────┬────────────┘      └───────────┬────────────┘ │
│              └────────────┬───────────────────┘             │
│                    adaptive blend (tanh-bounded weight)      │
└───────────────────────────┬───────────────────────────────┘
                             │ final_demand_forecast
                             ▼
┌─────────────────────────────────────────────────────────┐
│               PriceElasticityEngine (NumPy)               │
│  DPI = w1·z(velocity) + w2·z(intent) − w3·z(stock)        │
│         − w4·z(competitor_delta)                           │
│  price_multiplier = 1 + max_fluct · tanh(k · DPI)          │
│  clipped to guardrails: [-20%, +20%] of base_price          │
└───────────────────────────┬───────────────────────────────┘
                             ▼
                     dynamic_price (FastAPI JSON response)
```

**Endpoints** (`server.py`):

| Endpoint | Description |
|---|---|
| `GET /predict-demand?product_id=SKU_1001` | Ensemble demand forecast, using live click-stream velocity |
| `GET /get-dynamic-price?product_id=SKU_1001` | Elasticity-adjusted price, bounded ±20% |

## 🧮 The Math, Briefly

**Exponential Smoothing (Holt's linear trend), pure NumPy:**

```
L_t = α·y_t + (1−α)·(L_{t-1} + T_{t-1})      (level)
T_t = β·(L_t − L_{t-1}) + (1−β)·T_{t-1}      (trend)
F_{t+h} = L_t + h·T_t                          (forecast)
```

**Adaptive blend weight** (more live traffic → trust the ML branch more):

```
confidence(events) = 1 / (1 + e^(−k·(events − midpoint)))
w_baseline = clip(0.85 − 0.70·confidence, 0.15, 0.85)
```

**Demand Pressure Index → bounded price multiplier:**

```
DPI = clip(w1·z(velocity) + w2·z(intent) − w3·z(stock) − w4·z(competitor_delta), −1, 1)
multiplier = 1 + 0.20 · tanh(1.5 · DPI)
```

`tanh` saturates smoothly, so the ±20% guardrail acts as a soft physical
limit rather than a hard cutoff that would cause price discontinuities —
and the function is most sensitive to change near normal demand
conditions, mirroring real elasticity-of-demand curves.

## 🚀 Setup (3 steps, fully local)

```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Run the server (trains the ensemble on startup, then launches
#    the live event simulator as a background thread)
python server.py

# 3. Hit the endpoints
curl "http://localhost:8000/predict-demand?product_id=SKU_1001"
curl "http://localhost:8000/get-dynamic-price?product_id=SKU_1001"
```

Interactive API docs (Swagger UI) are auto-generated by FastAPI at
`http://localhost:8000/docs`.

To reproduce every metric quoted above:

```bash
python benchmark.py
```

## ⚠️ Known Limitations (intentionally out of scope)

- Trained on synthetic data calibrated to plausible retail patterns, not
  real transaction logs — swap the data source in `model.py` for
  production use.
- Single-process, in-memory state — by design, to stay local-first and
  infra-free. A production deployment would persist `world_state` and the
  event buffer externally (e.g. Redis) for multi-instance consistency.
- The GradientBoostingRegressor is retrained only at process startup;
  there's no online/incremental learning loop yet.
